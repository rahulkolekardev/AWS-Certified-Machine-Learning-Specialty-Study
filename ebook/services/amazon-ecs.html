Amazon Elastic Container Service (ECS)

Amazon ECS is a fully managed container orchestration service that runs Docker containers on EC2 instances or AWS Fargate. For ML, it can host microservices, feature pipelines, and inference services as containerized applications.

Key features

  • Task and service abstractions for running containers.
  • Support for EC2 launch type or serverless Fargate launch type.
  • Integration with ALB/NLB, CloudWatch, IAM, and ECR.
  • Service Auto Scaling based on metrics.
  • Supports GPU instances for containerized ML workloads.

Where it fits in ML pipelines

  • Hosting ML-powered microservices (for example, REST APIs that call SageMaker or Bedrock).
  • Running containerized feature engineering and ETL tasks.
  • Serving custom inference containers when SageMaker is not used.
  • Executing supporting services such as message brokers or monitoring agents in ML platforms.

CLI and SDK examples

Run a task (CLI)

aws ecs run-task \
  --cluster ml-cluster \
  --launch-type FARGATE \
  --task-definition ml-task-def \
  --network-configuration awsvpcConfiguration={subnets=[subnet-123],securityGroups=[sg-123],assignPublicIp=DISABLED}

List services (Python, boto3)

import boto3

ecs = boto3.client("ecs")
resp = ecs.list_services(cluster="ml-cluster")
for arn in resp.get("serviceArns", []):
    print(arn)

Best practices for ML workloads

  • Use Fargate for simpler operations when you do not need node-level control.
  • For GPU-based inference, use EC2 launch type with GPU instances.
  • Integrate with Application Load Balancers for HTTP-based inference APIs.
  • Keep containers stateless and externalize state in databases or caches.

Exam tips

  • ECS vs. EKS: ECS is AWS-native and simpler; EKS is Kubernetes-based.
  • For serverless containers, pair ECS with Fargate.
  • Use ECS or EKS when exam scenarios mention container orchestration rather than ML-specific platforms.
