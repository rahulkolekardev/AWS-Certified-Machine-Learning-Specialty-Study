AWS Fargate

AWS Fargate is a serverless compute engine for containers that works with ECS and EKS. It lets you run containers without managing servers or clusters, paying only for the vCPU and memory resources used.

Key features

  • No need to provision or manage EC2 instances for containers.
  • Task-level isolation and automatic scaling.
  • Works with ECS tasks and EKS pods (Fargate profiles).
  • Integrated with VPC networking, IAM, and CloudWatch.
  • Suitable for many CPU-bound inference and microservice workloads.

Where it fits in ML pipelines

  • Running lightweight inference services or ML-powered microservices.
  • Hosting auxiliary services (feature APIs, web frontends, schedulers) in ML architectures.
  • Prototyping containerized ML workloads without managing node fleets.

CLI and SDK examples

Fargate is configured through ECS or EKS; see their pages for examples. You select the Fargate launch type or Fargate profiles when defining tasks or pods.

Best practices for ML workloads

  • Use Fargate for CPU-bound microservices; GPU workloads generally require EC2-based nodes.
  • Right-size vCPU and memory; overprovisioning directly impacts cost.
  • Forward logs to CloudWatch and use autoscaling policies based on utilization.
  • Use private subnets and security groups for network isolation.

Exam tips

  • Fargate is “serverless containers,” often paired with ECS or EKS in exam diagrams.
  • Use Fargate when you explicitly want to avoid managing EC2 clusters for containers.
  • For GPU or very specialized node configurations, EC2-based containers remain necessary.
