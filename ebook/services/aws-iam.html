AWS Identity and Access Management (IAM)

AWS IAM enables you to manage access to AWS services and resources securely. It is foundational for ML workloads, controlling which users, roles, and services can access data, training infrastructure, and model endpoints.

Key features

  • Users, groups, roles, and policies for identity-based access control.
  • Resource-based policies for services like S3 and Lambda.
  • Fine-grained permissions with condition keys.
  • Federation with identity providers (IdPs) using SAML or OIDC.
  • IAM Access Analyzer and policy validation tools.

Where it fits in ML pipelines

  • Defining execution roles for SageMaker training jobs, endpoints, and pipelines.
  • Granting least-privilege access to S3 buckets, data catalogs, and databases used in ML.
  • Controlling who can deploy or modify models in production.
  • Integrating corporate SSO to manage ML platform access.

CLI and SDK examples

Create a role (CLI)

aws iam create-role \
  --role-name SageMakerExecutionRole \
  --assume-role-policy-document file://trust-policy.json

Attach a policy (Python, boto3)

import boto3

iam = boto3.client("iam")
iam.attach_role_policy(
    RoleName="SageMakerExecutionRole",
    PolicyArn="arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess",
)

Best practices for ML workloads

  • Apply least-privilege principles for all ML-related roles and users.
  • Use separate roles for training, inference, and data engineering workloads.
  • Limit direct user access to data; prefer service roles and federated access.
  • Regularly review and audit IAM policies, especially around data access.

Exam tips

  • IAM is central to any question about who can access which ML resources and data.
  • Know the difference between identity-based and resource-based policies.
  • Understand how SageMaker, Bedrock, and other services assume IAM roles on your behalf.
