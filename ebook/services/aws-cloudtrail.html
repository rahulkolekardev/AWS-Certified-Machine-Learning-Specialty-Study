AWS CloudTrail

AWS CloudTrail records API calls and account activity across your AWS environment. It is crucial for auditing, compliance, and forensic analysis, including tracking changes and access related to ML resources.

Key features

  • Event history for management events (control plane operations).
  • Data events for object-level access in services like S3 and Lambda.
  • Trails that deliver logs to S3 and optionally to CloudWatch Logs.
  • CloudTrail Lake for querying events centrally.
  • Integration with security and compliance tooling.

Where it fits in ML pipelines

  • Auditing who created, modified, or deleted ML endpoints, training jobs, and data stores.
  • Investigating security incidents involving access to training data or model artifacts.
  • Proving compliance and traceability in regulated ML environments.
  • Feeding security analytics systems with ML-related activity logs.

CLI and SDK examples

Look up events (CLI)

aws cloudtrail lookup-events \
  --lookup-attributes AttributeKey=EventSource,AttributeValue=sagemaker.amazonaws.com

Look up events (Python, boto3)

import boto3

ct = boto3.client("cloudtrail")

resp = ct.lookup_events(
    LookupAttributes=[{
        "AttributeKey": "EventSource",
        "AttributeValue": "sagemaker.amazonaws.com",
    }],
    MaxResults=10,
)
for event in resp.get("Events", []):
    print(event["EventName"], event["EventTime"])

Best practices for ML workloads

  • Enable CloudTrail in all Regions and for all accounts in an organization.
  • Store CloudTrail logs in a dedicated, locked-down S3 bucket.
  • Use CloudTrail Lake or log analytics tools to query ML-related events efficiently.
  • Combine CloudTrail with CloudWatch logs and metrics for full operational visibility.

Exam tips

  • CloudTrail is for API call history and governance, not for performance monitoring.
  • Scenarios involving “who did what, when” or investigations rely on CloudTrail.
  • Know the difference between management events (control plane) and data events.
