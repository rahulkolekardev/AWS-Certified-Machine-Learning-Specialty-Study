Amazon Elastic File System (EFS)

Amazon EFS is a fully managed, elastic NFS file system for use with AWS compute services. For ML, EFS provides a shared file system that can be mounted by multiple EC2 instances, ECS tasks, or EKS pods simultaneously.

Key features

  • POSIX-compliant NFS file system accessible from many instances.
  • Automatic growth and shrinkage with usage.
  • Standard and One Zone storage classes, plus infrequent access tiers.
  • Encryption at rest and in transit.
  • Integration with VPC security groups and mount targets.

Where it fits in ML pipelines

  • Sharing training data, code, or checkpoints across multiple training workers.
  • Storing common libraries or datasets for a cluster of ML instances.
  • Feeding inference workloads that run across many containers needing shared files.

CLI examples

Create a file system (CLI)

aws efs create-file-system \
  --creation-token ml-efs \
  --performance-mode generalPurpose

Best practices for ML workloads

  • Use EFS for shared, file-based workloads that need concurrent access.
  • For high-throughput local storage per instance, prefer EBS or instance store.
  • Place mount targets in subnets close to your compute resources to reduce latency.
  • Leverage lifecycle management to move cold data to infrequent access tiers.

Exam tips

  • EFS is the shared file system service; EBS is single-instance block storage.
  • For HPC or highly parallel training needing shared POSIX semantics, EFS or FSx may be referenced.
  • Remember that EFS is regional and can be accessed from multiple AZs in a Region.
