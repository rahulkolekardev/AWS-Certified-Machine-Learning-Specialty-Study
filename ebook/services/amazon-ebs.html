Amazon Elastic Block Store (EBS)

Amazon EBS provides block-level storage volumes for EC2 instances. For ML, it is used as durable, high performance storage for training data, intermediate artifacts, and model files attached to compute instances.

Key features

  • Multiple volume types (gp3, io2, st1, sc1) optimized for performance and cost.
  • Point-in-time snapshots stored in S3.
  • Encryption at rest with KMS.
  • Provisioned IOPS for high-performance workloads.
  • Ability to detach and reattach volumes across compatible instances.

Where it fits in ML pipelines

  • Providing local storage for EC2-based ML training and inference.
  • Storing temporary datasets, caches, or checkpoints.
  • Supporting high I/O workloads such as on-prem-to-cloud migrations or heavy preprocessing.

CLI examples

Create a volume (CLI)

aws ec2 create-volume \
  --size 200 \
  --volume-type gp3 \
  --availability-zone us-east-1a

Best practices for ML workloads

  • Use gp3 or io2 for latency-sensitive, high-throughput ML training workloads.
  • Snapshot volumes before risky experiments for quick rollback.
  • Use EBS-optimized instances and appropriate file systems for performance.
  • Delete unused volumes and snapshots to control costs.

Exam tips

  • EBS is per-instance block storage, not shared across instances (unlike EFS or FSx).
  • Performance tuning questions may involve volume type and provisioned IOPS selection.
  • For massive datasets and data lakes, S3 is usually the right storage layer, not EBS.
