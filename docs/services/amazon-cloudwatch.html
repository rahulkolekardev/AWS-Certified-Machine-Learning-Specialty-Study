Amazon CloudWatch

Amazon CloudWatch is a monitoring and observability service that collects metrics, logs, and events from AWS resources and applications. For ML, it is used to monitor training jobs, endpoints, pipelines, and underlying infrastructure.

Key features

  • Metrics, logs, and alarms for AWS services and custom applications.
  • Dashboards for real-time visualization of metrics.
  • Log Insights for querying and analyzing log data.
  • Events and rules for responding to state changes.
  • ServiceLens and cross-service tracing (with X-Ray) for deeper visibility.

Where it fits in ML pipelines

  • Tracking training job resource usage, errors, and completion metrics.
  • Monitoring inference latency, error rates, and throughput for endpoints.
  • Setting alarms on key business or model quality metrics emitted as custom metrics.
  • Capturing logs from SageMaker, Bedrock, ECS/EKS, and Batch jobs.

CLI and SDK examples

Put a custom metric (CLI)

aws cloudwatch put-metric-data \
  --namespace MLApp \
  --metric-data 'MetricName=Latency,Value=45,Unit=Milliseconds'

Get metrics (Python, boto3)

import boto3

cw = boto3.client("cloudwatch")

resp = cw.list_metrics(Namespace="AWS/SageMaker")
for m in resp.get("Metrics", [])[:5]:
    print(m["MetricName"], m.get("Dimensions", []))

Best practices for ML workloads

  • Emit domain-specific metrics (for example, prediction error rate) in addition to infrastructure metrics.
  • Use dashboards to give product and operations teams a shared view of model health.
  • Configure alarms with sensible thresholds and notification channels (SNS, Slack, etc.).
  • Leverage log retention policies and metric filters to manage costs and focus on important signals.

Exam tips

  • CloudWatch is for monitoring and alerting; CloudTrail is for auditing API activity.
  • Exam questions about latency, errors, and autoscaling almost always involve CloudWatch metrics and alarms.
  • Remember that many ML services automatically publish metrics and logs to CloudWatch.
