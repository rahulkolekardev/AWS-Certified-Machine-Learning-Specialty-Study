Amazon FSx

Amazon FSx provides managed file systems built on popular file storage technologies such as Windows File Server, Lustre, NetApp ONTAP, and OpenZFS. In ML workloads, FSx for Lustre is particularly relevant for high-performance, POSIX-compliant storage.

Key features (FSx for Lustre)

  • High-throughput, low-latency file system optimized for compute-intensive workloads.
  • Integration with S3 for linking file systems to S3 buckets.
  • Multiple deployment options (scratch and persistent).
  • POSIX-compliant semantics for Linux-based compute clusters.
  • Scales to many GB/s of throughput and millions of IOPS.

Where it fits in ML pipelines

  • Large-scale training jobs on EC2 or EKS that need shared, high-performance storage.
  • HPC-style ML workloads that benefit from Lustre’s parallel I/O.
  • Speeding up access to training datasets stored in S3 via linked FSx for Lustre file systems.

CLI examples

Create an FSx for Lustre file system (CLI)

aws fsx create-file-system \
  --file-system-type LUSTRE \
  --storage-capacity 1200 \
  --subnet-ids subnet-1234567890abcdef0

Best practices for ML workloads

  • Use FSx for Lustre for high-performance, shared file access across many training nodes.
  • Link FSx for Lustre to S3 buckets for bursty training workloads that read large datasets.
  • Select scratch vs. persistent deployment types based on durability and lifespan needs.
  • Ensure your EC2 instances are in the same AZ and VPC as the FSx file system for best performance.

Exam tips

  • FSx for Lustre is highlighted when high-throughput, shared POSIX file storage for training is required.
  • EFS focuses on general-purpose POSIX sharing; FSx for Lustre focuses on high-performance workloads.
  • Look for mentions of “HPC training cluster” or “many-node deep learning job.”
