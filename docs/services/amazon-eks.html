Amazon Elastic Kubernetes Service (EKS)

Amazon EKS is a managed Kubernetes service that makes it easy to run Kubernetes clusters on AWS without managing the control plane. Many teams use EKS to standardize ML infrastructure around Kubernetes-native tools and patterns.

Key features

  • Managed Kubernetes control plane with automatic scaling and patching.
  • Worker nodes on EC2 or Fargate.
  • Integration with IAM, VPC, and CloudWatch for security and observability.
  • Support for GPU nodes for ML training and inference.
  • Compatibility with the broader Kubernetes ecosystem (Helm, operators, etc.).

Where it fits in ML pipelines

  • Running Kubernetes-native ML platforms such as Kubeflow or Ray on K8s.
  • Hosting microservices and inference workloads that rely on Kubernetes patterns.
  • Portability across environments (on-premises and AWS) using the same control plane tooling.
  • Shared ML infrastructure for large organizations standardizing on Kubernetes.

CLI and SDK examples

Describe a cluster (CLI)

aws eks describe-cluster --name ml-eks-cluster

List clusters (Python, boto3)

import boto3

eks = boto3.client("eks")
resp = eks.list_clusters()
print(resp.get("clusters", []))

Best practices for ML workloads

  • Use managed node groups and autoscaling for balancing cost and performance.
  • Isolate workloads in namespaces and use network policies for security.
  • Consider EKS + Fargate for simpler operations of small services; use EC2 nodes for GPU-heavy jobs.
  • Integrate with ECR, IAM roles for service accounts, and VPC CNI for robust networking.

Exam tips

  • EKS is chosen when Kubernetes portability and ecosystem integration are emphasized.
  • For simpler container needs without Kubernetes, ECS is usually preferred in exam answers.
  • Managed control plane and Kubernetes-native tooling are key differentiators.
