Amazon Virtual Private Cloud (VPC)

Amazon VPC lets you provision logically isolated virtual networks in AWS. For ML workloads, VPCs provide network isolation, control over routing, and private connectivity between data stores, compute resources, and managed ML services.

Key features

  • Subnets, route tables, internet and NAT gateways.
  • Security groups and network ACLs for traffic control.
  • VPC endpoints for private connectivity to AWS services.
  • Peering, Transit Gateway, and VPN/Direct Connect for hybrid architectures.
  • Flow logs for network traffic visibility.

Where it fits in ML pipelines

  • Isolating training and inference environments in private subnets.
  • Providing private access to S3, Bedrock, SageMaker, and other services via VPC endpoints.
  • Connecting on-premises data centers to AWS-based ML stacks.
  • Implementing network segmentation between dev, test, and production ML environments.

CLI examples

Create a VPC (CLI)

aws ec2 create-vpc --cidr-block 10.0.0.0/16

Best practices for ML workloads

  • Place ML endpoints and training instances in private subnets; use load balancers or endpoints for access.
  • Use interface VPC endpoints to keep traffic to AWS APIs (including S3 and SageMaker) inside the AWS network.
  • Segment networks by environment and sensitivity level, especially for regulated workloads.
  • Monitor VPC Flow Logs and enforce least-privilege network access via security groups.

Exam tips

  • VPC is central to questions about private ML endpoints, data exfiltration controls, and hybrid connectivity.
  • VPC endpoints are key for private S3 and Bedrock/SageMaker access from private subnets.
  • Remember that security groups are stateful, while network ACLs are stateless.
